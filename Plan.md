# Архитектурное решение и план работ
## Система учёта сменных событий, инцидентов и коммуникации инженерной команды

---

## 1. Обзор системы

**Наименование:** Система учёта сменных событий, инцидентов и внутренних коммуникаций инженерной службы

**Назначение:** Внутренняя веб-система для фиксации событий смены, учёта инцидентов, контроля выполнения работ, ведения истории действий персонала, организации коммуникации между сменными сотрудниками и руководством, а также накопления знаний через модуль Wiki.

## 5. Схема базы данных

### Сущности

```
users
  id, username, full_name, password_hash, role, is_active, created_at

shifts
  id, author_id → users, shift_type, started_at, closed_at,
  equipment_status, completed_works, open_issues, notes, is_closed

incidents
  id, author_id → users, assignee_id → users,
  title, description, category, priority, status,
  equipment_ref, created_at, closed_at, resolution

comments
  id, author_id → users, entity_type, entity_id,
  body, created_at

audit_log
  id, user_id → users, action_type, entity_type, entity_id,
  old_value (jsonb), new_value (jsonb), created_at

messages
  id, author_id → users, body, needs_attention, is_read, created_at

wiki_sections
  id, parent_id → wiki_sections, title

wiki_articles
  id, author_id → users, section_id → wiki_sections,
  title, content (Markdown), tags, created_at, updated_at

wiki_article_versions
  id, article_id → wiki_articles, author_id → users,
  content, created_at

equipment
  id, name, location, description
```

`Новый` → `В работе` → `Ожидает` → `Решён` → `Закрыт` / `Отклонён`

### Приоритеты инцидентов

`Низкий` / `Средний` / `Высокий` / `Критический`

## 6. Ключевые архитектурные решения

### 6.1. Аутентификация

**Рекомендация:** сессионная авторизация через Redis (не JWT).

Причины:
- Мгновенный отзыв сессии при блокировке сотрудника
- Не нужно управлять refresh-токенами
- Проще реализация на внутреннем инструменте без мобильного клиента

> Если в будущем планируется мобильный клиент — перейти на JWT с refresh-токенами.

### 6.2. Audit log

Реализовать через **FastAPI middleware + SQLAlchemy event listeners**, а не ручными вставками в каждый эндпоинт. Это гарантирует, что ни одно действие не будет пропущено.

Логируемые события:
- Вход пользователя в систему
- Создание / редактирование записи смены, закрытие смены
- Создание инцидента, изменение статуса, смена ответственного
- Добавление комментариев, закрытие инцидента
- Создание / редактирование / удаление статей Wiki

### 6.3. Wiki: версионирование

Хранить полный snapshot содержимого статьи при каждом сохранении в таблице `wiki_article_versions`. Diff вычислять на фронтенде через библиотеку `diff`. Отдельный git-движок не нужен.

### 6.4. Полнотекстовый поиск

Использовать **PostgreSQL `tsvector` + `GIN`-индекс** — покрывает задачу без Elasticsearch.

```sql
-- Пример индекса для полнотекстового поиска по статьям
CREATE INDEX idx_wiki_fts ON wiki_articles
  USING GIN (to_tsvector('russian', title || ' ' || content));
```

### 6.5. Markdown-редактор для Wiki

Рекомендуемые библиотеки (лёгкие, без тяжёлых зависимостей):
- `@uiw/react-md-editor`
- `milkdown`

Хранить в БД исходный Markdown. Preview рендерить на фронтенде через `remark`.

## 10. Критерии приёмки MVP

- [ ] Пользователи могут авторизоваться под своими учётными записями
- [ ] Доступно создание, просмотр и редактирование записей смен
- [ ] Доступно создание и ведение карточек инцидентов со статусами
- [ ] Сохраняется история изменений (audit log)
- [ ] Доступна фильтрация и поиск по инцидентам и сменам
- [ ] Руководитель может просматривать сводную информацию и служебные сообщения
- [ ] Система стабильно работает в локальной сети на 10–30 пользователей
