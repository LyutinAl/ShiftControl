# Postmortem: Недоступность API (ServiceDown)

## Дата и время
- **Начало:** 2026-05-12 10:22:42
- **Обнаружение:** 2026-05-12 10:23:49 (TTD: 1 мин 7 сек)
- **Восстановление:** 2026-05-12 10:25:12 (TTR: 2 мин 30 сек)

## Описание
Сервис API ShiftControl стал недоступен из-за аварийной остановки контейнера `shiftcontrol_backend`.
Prometheus зафиксировал недоступность endpoint `/metrics` на `host.docker.internal:8000`,
алерт `ServiceDown` сработал через 1 минуту и доставил уведомление на почту дежурного инженера.

## Влияние
- **Пользователи:** все активные сессии — API недоступен полностью
- **Затронутые запросы:** ~100% входящего трафика за период 10:22–10:25
- **Сервисы:** FastAPI backend (аутентификация, смены, инциденты, сообщения)
- **Бизнес:** невозможность создания и закрытия смен, потеря входящих запросов за 2 мин 30 сек

## Root Cause (5 Why)

1. **Почему API был недоступен?** — Контейнер `shiftcontrol_backend` остановился.
2. **Почему контейнер остановился?** — Процесс завершился с ненулевым кодом выхода (симуляция: ручная остановка).
3. **Почему контейнер не перезапустился автоматически?** — Политика `restart: unless-stopped` не перезапускает контейнеры, остановленные вручную командой `docker stop`.
4. **Почему не было превентивного мониторинга состояния процесса?** — Healthcheck настроен на уровне Terraform, но Prometheus не получал сигнала о деградации до полной остановки.
5. **Почему не было автоматического восстановления?** — Оркестратор уровня кластера (Kubernetes) отсутствует; используется одиночный Docker-контейнер без self-healing.

## Что сработало хорошо
- Алерт `ServiceDown` сработал ровно через 1 минуту после потери доступности — в рамках SLO по TTD.
- Email-уведомление доставлено немедленно после срабатывания алерта.
- Дашборд Grafana (API Golden Signals) сразу показал падение Request Rate до нуля.
- Логи контейнера в дашборде Container Logs зафиксировали момент остановки.

## Что улучшить

- Отсутствует алерт на количество перезапусков контейнера (`restart_count`) — добавить как раннее предупреждение.
- Runbook в теле письма содержит только ссылку-заглушку — написать конкретные шаги диагностики.
- Нет алерта на аномально низкий Request Rate (близкий к нулю, но не точно нулевой) — поможет поймать частичную деградацию.
