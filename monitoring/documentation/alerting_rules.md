# Обоснование правил алертов — ShiftControl

## Таблица алертов

| Алерт         | Severity | For  | Порог          | Команда        |
|---------------|----------|------|----------------|----------------|
| HighErrorRate | critical | 5m   | 5xx > 5%       | backend        |
| HighLatency   | warning  | 10m  | P95 > 500ms    | backend        |
| ServiceDown   | critical | 1m   | up == 0        | backend        |
| HighCPUUsage  | warning  | 5m   | CPU > 80%      | infrastructure |

---

## HighErrorRate

**Выражение:**
```promql
(
  sum(rate(http_requests_total{status_code=~"5.."}[5m]))
  /
  sum(rate(http_requests_total[5m]))
) * 100 > 5
```

**Почему порог 5%?**
- Нормальный уровень 5xx для корректно работающего API — менее 0.5% (ошибки БД, сетевые таймауты)
- Порог 5% даёт 10-кратный запас от нормы: срабатывает только при реальной деградации, а не точечных сбоях
- Ниже 5% риск ложных срабатываний при кратковременных ошибках зависимостей

**Почему `for: 5m`?**
- Фильтрует кратковременные всплески (деплой, перезапуск БД-соединений)
- Гарантирует устойчивую проблему, а не шум

**Необходимые действия:**
1. Открыть дашборд API Golden Signals → панель "Request Rate by Status" (определить затронутые endpoints)
2. Открыть Container Logs в Grafana, фильтр по `shiftcontrol_backend`, искать `ERROR` / `traceback`
3. Проверить последние деплои (`git log --since="1 hour ago"`)
4. Проверить доступность PostgreSQL: `docker exec shiftcontrol_db pg_isready`
5. При необходимости — откатить последний деплой

**Эскалация:**
- Critical → Email на дежурного инженера (немедленно)

## HighLatency

**Выражение:**
```promql
histogram_quantile(0.95,
  rate(http_request_duration_seconds_bucket[5m])
) > 0.5
```

**Почему P95 и порог 500ms?**
- P95 отражает опыт 95% пользователей, не зависит от единичных медленных запросов
- 500ms — граница воспринимаемой отзывчивости UI (Nielsen Norman Group: > 500ms пользователь замечает задержку)
- P50 или mean скрыли бы деградацию для «хвоста» запросов

**Почему `for: 10m`?**
- Латентность естественно возрастает при холодном старте, GC-паузах, пиковой нагрузке
- 10 минут гарантируют, что это системная проблема, а не временный пик

**Необходимые действия:**
1. Открыть дашборд → "Latency Percentiles" (сравнить P50/P95/P99, определить масштаб)
2. Проверить метрику `shiftcontrol_db_query_duration_seconds` — часто корень проблемы в медленных запросах к БД
3. Проверить `node_cpu_seconds_total` и `node_memory_MemAvailable_bytes` — деградация из-за ресурсов
4. Проверить нет ли `N+1` запросов в логах (`SELECT` в цикле)

**Эскалация:**
- Warning → Email на команду backend

## ServiceDown

**Выражение:**
```promql
up{job="api"} == 0
```

**Почему `up == 0`?**
- Prometheus не может получить метрики с endpoint — сервис либо упал, либо недоступен по сети
- Самый надёжный способ детектировать полную недоступность: не зависит от содержимого метрик

**Почему `for: 1m`?**
- Минимальное время, чтобы исключить кратковременный рестарт контейнера (restart: unless-stopped)
- После 1 минуты недоступность точно не является плановым рестартом

**Необходимые действия:**
1. Проверить статус контейнера: `docker ps | grep shiftcontrol_backend`
2. Посмотреть последние логи: `docker logs shiftcontrol_backend --tail 50`
3. При статусе Exited — проверить код выхода: `docker inspect shiftcontrol_backend | grep ExitCode`
4. Перезапустить: `docker start shiftcontrol_backend`
5. Если не стартует — проверить доступность БД и Redis

**Эскалация:**
- Critical → Email на дежурного (немедленно)

## HighCPUUsage

**Выражение:**
```promql
(
  1 - avg(rate(node_cpu_seconds_total{mode="idle"}[5m]))
) * 100 > 80
```

**Почему порог 80%?**
- До 70% — нормальная нагрузка под пиком, не влияет на латентность
- 80–90% — начинается деградация: увеличивается время ответа, растут очереди
- 100% — полное насыщение, ошибки таймаутов
- Запас 20% до максимума позволяет успеть среагировать до деградации пользовательского опыта

**Почему `avg` по всем ядрам?**
- Однопоточные процессы могут насытить одно ядро, не трогая остальные
- Для данного сервиса важна общая нагрузка хоста, а не отдельного ядра

**Почему `for: 5m`?**
- Кратковременные всплески (GC, индексирование, cron) — норма
- 5 минут устойчивой нагрузки сигнализируют о реальной проблеме

**Необходимые действия:**
1. Открыть дашборд Infrastructure → CPU Usage (определить тренд)
2. Проверить какой процесс потребляет CPU: `docker stats`
3. Если нагрузка от `shiftcontrol_backend` — проверить нет ли неоптимальных запросов или бесконечных циклов
4. Рассмотреть горизонтальное масштабирование или rate limiting

**Эскалация:**
- Warning → Email на команду infrastructure
